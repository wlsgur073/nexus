## 1. 사용자 역할 정의

### 1.1 역할별 특성

#### 신청자 (Requester)

- **대상**: 비즈니스 분석가, 개발자, 데이터 엔지니어
- **핵심 니즈**: 기존 표준을 빠르게 검색하고, 필요 시 신규/변경/삭제 신청을 최소 단계로 제출하는 것
- **주요 시나리오**:
  - 표준용어를 검색하여 물리 컬럼명 확인
  - 존재하지 않는 표준단어를 신규 신청
  - 기존 표준의 정의를 변경 신청
  - 불필요한 표준의 삭제 신청 (영향도 평가 포함)
  - 피드백 받은 신청 건을 수정/재신청
- **빈도**: 일 1-5회 검색, 주 1-3회 신청
- **Pain Point**: 검색과 신청이 분리되어 화면 전환이 잦음, 신청 상태 추적이 불편

#### 검토/승인자 (Reviewer/Approver)

- **대상**: 표준 관리자, 데이터 거버넌스 리더
- **핵심 니즈**: 대기 중인 신청 건을 효율적으로 검토하고 승인/반려/피드백 처리하는 것
- **주요 시나리오**:
  - 승인 대기 건수 확인 및 우선순위 파악
  - 변경 신청의 전/후 비교 확인
  - 일괄 승인/반려 처리
  - 검토 의견(피드백) 전달
- **빈도**: 일 3-10회 승인 처리
- **Pain Point**: 상세 확인을 위해 별도 화면 이동 필요, 맥락 전환 비용 높음

#### 표준 관리자 (STD_MANAGER)

- **대상**: 거버넌스 매니저, 데이터 품질 리더
- **핵심 니즈**: 조직 전체 표준 품질을 모니터링하고, 교차 도메인 일관성을 유지하는 것
- **주요 시나리오**:
  - 거버넌스 포털에서 준수율 모니터링
  - 검증 대시보드에서 위반 항목 추적
  - 공통코드 관리 및 거버넌스
  - 감사 추적 조회
- **빈도**: 일 1-2회 모니터링, 주 1-2회 집중 리뷰
- **Pain Point**: 분산된 모니터링 지표, 위반 → 시정 연결 어려움

#### 시스템관리자 (System Administrator)

- **대상**: IT 운영팀, 플랫폼 관리자
- **핵심 니즈**: 사용자 계정, 권한, 시스템 코드, DB 연결 등 운영 설정을 관리하는 것
- **주요 시나리오**:
  - 사용자 계정 생성/비활성화
  - 역할별 메뉴 권한 설정
  - 시스템 코드 관리 (보호 코드 정책)
  - DB 연결/SSH 터널링 설정
- **빈도**: 주 1-2회 (이벤트 기반)
- **Pain Point**: 관리 화면 접근 빈도 낮아 UI 숙지가 어려움

#### 조회전용 (Read-only User)

- **대상**: 인증된 제한 권한 사용자
- **핵심 니즈**: 표준 정보를 검색하고 참조하는 것
- **주요 시나리오**:
  - 표준용어 검색 및 물리명 확인
  - 공통코드 조회
  - 검증 대시보드 현황 확인
- **빈도**: 일 1-3회 검색
- **Pain Point**: 사용 가능 기능 범위를 명확히 인지하기 어려움

## 2. 사용자 여정 맵 (User Journey Map)

### 2.1 신청자 — 표준 검색 → 변경 신청 플로우

```markdown
[로그인]
│
▼
[신청자 대시보드]──────────────────────────────────────────┐
│ │
│ "승인대기 2건" 알림 확인 │
│ "피드백대기 1건" → 클릭 시 해당 신청 Sheet 열림 │
│ │
▼ │
[통합 표준 탐색기] │
│ │
│ 탭: 표준용어 선택 │
│ 검색: "고객번호" 입력 │
│ AI Data Butler: "고객번호(CUST_NO)" 자동완성 추천 │
│ │
▼ │
[검색 결과 행 클릭] │
│ │
▼ │
[표준 상세 Sheet 열림 (화면 우측 슬라이드)] │
│ │
│ 기본 정보 확인 │
│ "편집" 버튼 클릭 → 인라인 편집 모드 전환 │
│ 정의 필드 수정 │
│ "변경 신청 제출" → 확인 Dialog │
│ │
▼ │
[Toast: "변경 신청이 제출되었습니다 (REQ-2026-0343)"] │
│ │
└────────────────────────────────────────────────────────┘
대시보드의 "내 신청" 위젯에 신규 건 반영
```

### 2.2 신청자 — 신규 표준 등록 플로우

```markdown
[통합 표준 탐색기]
│
│ 검색: "결제수단코드" → 결과 없음
│ AI Data Butler: "유사 표준 3건 발견" 패널 표시
│ "그래도 신규 신청" 버튼 클릭
│
▼
[신규 표준 신청 화면]
│
│ 유형 선택: 표준용어
│ 폼 입력: 용어명*, 도메인유형*, 인포타입*, 정의*
│ 물리명 자동 생성: PAY_MTHD_CD
│ AI Data Butler 미니패널: 유사 표준 3건 표시
│
│ ┌─ [초안 저장] ─── Draft로 저장, 나중에 제출 가능
│ │
│ └─ [신청 제출] ─── 바로 신청 (PENDING 상태)
│
▼
[Toast: "신청이 제출되었습니다"]
[대시보드로 자동 이동 또는 현재 화면 유지 선택]
```

### 2.3 승인자 — 승인 처리 플로우

```markdown
[로그인]
│
▼
[승인자 대시보드]
│
│ "승인 대기 5건" KPI 카드 클릭
│ 또는 "미처리 12일 경과 1건" 긴급 알림 클릭
│
▼
[승인 워크벤치]
│
│ 좌: 신청 목록 (필터: 구분, 요청구분)
│ 우: 선택된 신청의 상세 패널
│
│ 상세 패널에서:
│ - 변경 비교 테이블 (현재값 vs 변경요청값, 하이라이트)
│ - 신청 사유 확인
│ - 인라인 코멘트: 특정 필드에 의견 작성
│ - 처리 이력 타임라인
│
│ ┌─ [승인] + 처리사유 입력 → 표준 원장 반영
│ ├─ [반려] + 반려사유 입력 → 신청자에게 알림
│ └─ [검토요청] + 피드백 내용 입력 → 신청자에게 수정 요청
│
│ 일괄 처리: 체크박스 선택 → "일괄 승인" / "일괄 반려"
│
▼
[Toast: "3건 일괄 승인 처리되었습니다"]
[대시보드 KPI 자동 갱신]
```

### 2.4 표준 관리자 — 품질 모니터링 플로우

```markdown
[거버넌스 포털]
│
│ 준수율 게이지 확인
│ "표준용어 준수율 79%" → 하락 확인
│
▼
[검증 대시보드] (사이드바 또는 링크로 이동)
│
│ 위반 127건 확인
│ "명명규칙 위반 54건" 클릭
│
▼
[검증 상세] (명명규칙 탭 활성)
│
│ 위반 항목 목록 확인
│ "payAmt → PAY_AMT 권장" 확인
│ 체크박스 선택 → "일괄 시정 신청"
│
▼
[변경 신청 자동 생성, Toast 확인]
[감사 추적에서 시정 이력 확인]
```

### 2.5 시스템관리자 — 사용자/권한 관리 플로우

```markdown
[사용자 관리]
│
│ "사용자 추가" 클릭 → Dialog
│ ID, 이름, 비밀번호, 역할, 부서, 이메일 입력
│ "저장" → 사용자 생성
│
▼
[권한 관리]
│
│ 역할 탭 선택 (시스템관리자 / 검토승인자 / 표준 관리자 / 신청자 / 조회전용)
│ 메뉴 트리에서 접근 허용 메뉴 선택
│ CRUD 체크박스 조정
│ "저장" → 권한 반영
```
